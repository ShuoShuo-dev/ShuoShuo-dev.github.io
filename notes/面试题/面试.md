# HTML & CSS & Vue & JS 前端知识点整理
## HTML
### 1. 常用的html标签有哪些
#### 基础文档结构标签（页面骨架）
- html：根标签，包裹所有页面内容；
- head：存页面元信息（如title、meta），不显示在页面；
- title：定义页面标题（浏览器标签栏显示，SEO 核心）；
- body：页面主体，所有可见内容的容器。

#### 文本排版标签（文本展示）
- h1-h6：标题标签（h1 权重最高，一个页面仅 1 个），自带层级和加粗样式；
- p：段落标签，自动换行且有默认间距；
- span：行内通用标签（无默认样式，包裹小段文本做样式 / 交互）；
- strong/em：语义化加粗 / 斜体（区别于纯样式的b/i）；
- br（换行）、hr（分隔线）：单标签，辅助文本排版。

#### 语义化区块标签（提升可读性 / SEO）
- header/nav/footer：页头 / 导航 / 页脚；
- main：页面核心内容（唯一）；
- section/article：独立内容模块 / 完整内容（如文章、产品板块）；
- aside：侧边栏；
- div：块级通用标签（无语义，仅用于布局划分）。

#### 列表标签（结构化列表内容）
- ul+li：无序列表（如导航菜单、商品列表）；
- ol+li：有序列表（如步骤说明）；
- dl+dt+dd：定义列表（如 FAQ、产品参数）。

#### 链接标签
- a：超链接，核心属性href（跳转地址），target="_blank"新窗口打开，也可做锚点跳转（href="#id"）。

#### 表单标签（收集用户信息）
- form：表单容器（action提交地址、method提交方式）；
- input：核心表单元素，type属性多态（text/password/radio/checkbox/button/submit/file 等）；
- textarea：多行文本输入、select+option：下拉选择框、label：绑定表单元素（提升交互）。

#### 多媒体标签
- img：图片标签（单标签），核心属性src（地址）、alt（加载失败替代文本，无障碍 + SEO）；
- audio/video：音频 / 视频标签，controls属性显示播放控件。

#### 表格标签（展示结构化数据）
- table：表格容器，搭配thead/tbody/tr/th/td使用（表头 / 主体 / 行 / 表头单元格 / 普通单元格）。

### 2. js可以放到header里面吗为啥不行，css为啥放到header里面在html加载前面也可以
#### 一、JS 可以放 header 里，但不推荐（核心原因）
1. 阻塞页面渲染：浏览器解析 HTML 是从上到下的，`<header>`里的 JS 会被优先执行，且 JS 执行时会暂停 HTML 解析和页面渲染（JS 可能操作 DOM，需保证执行时 DOM 已存在）。
   - 若 JS 放 header 且未做异步处理，会导致页面空白直到 JS 加载 / 执行完成，用户体验差；
   - 若 JS 中操作了还未解析的 DOM（如document.getElementById），会直接报错。
2. 补充：并非 “完全不行”，若 JS 是异步加载（async/defer）或仅定义函数（不执行 DOM 操作），放 header 也可，但日常开发中仍优先放`</body>`前。

#### 二、CSS 必须 / 推荐放 header 里（核心原因）
1. 避免页面 “闪屏”：浏览器渲染页面时，会先解析 CSS 生成 CSSOM（CSS 对象模型），再结合 DOM 生成渲染树。若 CSS 放页面底部，页面会先以默认样式渲染内容，等 CSS 加载后再重绘，出现 “样式闪烁”；
2. CSS 不阻塞 HTML 解析：CSS 加载时不会暂停 HTML 解析，仅阻塞渲染（合理设计），放 header 能让 CSS 尽早加载，页面渲染时直接应用样式，保证视觉一致性；
3. 性能优化：CSS 放 header 可被浏览器提前预加载，配合缓存策略提升加载效率。

### 3. 标题是什么写在哪
HTML 页面标题用 `<title>` 写，写在 `<head>` 里面。

### 4. 块元素和行内元素的区别
#### 一、布局表现（最核心）
1. 块元素：独占一行，默认宽度撑满父容器（100%），可手动设置宽高（width/height）；
   典型：div、p、h1-h6、ul/li、form 等。
2. 行内元素：不独占一行，多个行内元素可并排显示，默认宽高由内容决定，无法手动设置宽高；
   典型：span、a、strong、em、img（特殊行内替换元素，可设宽高）等。

#### 二、盒模型属性支持
1. 块元素：支持 margin、padding 的上下左右所有方向，border 也完全生效；
2. 行内元素：margin/padding 仅左右生效，上下不影响布局（视觉上可能显示但不占空间），border 上下会重叠到相邻行。

#### 三、嵌套规则
1. 块元素：可嵌套块元素、行内元素（特殊：p 标签不能嵌套 div 等块元素）；
2. 行内元素：只能嵌套行内元素，不能嵌套块元素（a 标签除外，可嵌套块元素）。

### 5. style、script、body 标签书写顺序
CSS（style 标签）最好写在 head 里面；script 标签建议写在 body 后面（文档末尾）；div、span 等页面元素写在 body 内部，整体建议顺序：`head`（含 style）→ `body`（含页面元素）→ `script`。

### 6. image标签中alt属性的作用
当图片因某种原因加载不出来的时候，在图片显示框里展示的说明文字。

### 7. 块级元素、行内元素、行内块元素的区别
- 块级标签：默认独占一行，可设置宽高、边距，例如div,p,h1-h6
- 行内元素：不换行，不可设置宽高，上下内外边距无效，例如span、a 
- 行内块元素：不换行，但能设置宽高、内外边距，例如img、button、input

### 8. HTML中的块级元素
`<div>`、`<p>`、`<h1>-<h6>`、`ul`、`ol`、`li`、`table`、`form`、`header`、`footer`、`hr`（水平线）。

### 9. Image图片元素的分类
行内块元素（不换行，可以设置宽高）。

## CSS
### 1. css是怎么布局的
- Flex 弹性布局：给父元素设`display: flex`，通过`justify-content`（水平对齐）、`align-items`（垂直对齐）、`flex-direction`（排列方向）等属性，快速实现元素的灵活排布（如居中、均分、自适应），适配响应式，是日常布局首选； 
- Grid 网格布局：给父元素设`display: grid`，按 “行 + 列” 二维网格划分区域，精准控制元素的行列位置和尺寸（如九宫格、复杂表单布局），适合复杂二维布局；

### 2. span如何设置宽高，span里面能放p标签不
#### span 设置宽高的方法
span 是行内元素，默认宽高由内容决定，无法直接通过 width/height 设置，需先改变其显示模式：
1. 最常用：设置`display: inline-block`（行内块），既保留并排显示特性，又可自定义宽高；
2. 备选：设置`display: block`（块元素），但 span 会独占一行，失去行内元素特性；
3. 特殊场景：设置`position: absolute/fixed`（脱离文档流），也可直接设置宽高（无需改 display）。

#### span 里能否放 p 标签
**不能**（不符合 HTML 语义和嵌套规则）：
- span 是行内元素，p 是块元素，HTML 规范中行内元素不能嵌套块元素；
- 即使强行写，浏览器会自动拆解 DOM 结构（把 p 标签提至 span 外），导致布局 / 语义混乱；
- 若需在 span 内嵌套内容，只能放行内元素（如 a、em、另一个 span）。

### 3. 一个div元素有五个子元素，怎么让他们头部对齐
- Grid 布局：父元素设`display: grid; align-items: start`，直接控制子元素顶部对齐，现代布局首选； 
- inline-block+vertical-align：子元素设`display: inline-block; vertical-align: top`，兼容性最好； 
- 表格布局：父元素`display: table`，子元素`display: table-cell; vertical-align: top`，适配类表格场景； 
- 浮动：子元素设`float: left`（父元素清浮动），浮动元素默认顶部对齐，传统布局常用。

### 4. CSS中怎么让一个元素水平垂直居中
**flex布局**（首选）：
```css
.parent {
  display: flex;
  justify-content: center;    /* 水平居中 */
  align-items: center;        /* 垂直居中 */
}
```

## Vue
### 1. 区分const,let,var这几个关键字
- var：全局作用域，允许后声明的覆盖先前的，存在变量提升、重复声明问题；
- let和const：块级作用域，无变量提升、不允许重复声明；
- const：更适合赋值给常量，声明时必须初始化，不能修改变量指向（对象/数组可修改内部属性）；
- 开发共识：const 优先，let 次之，var 不用。

### 2. 从前端角度区分后端的数据类型
#### 一、核心判断方法（通用逻辑）
前端拿到的后端数据本质是JSON 解析后的 JS 对象 / 数组 / 基本类型，主要通过以下方式区分：
1. 基础类型判断（字符串 / 数字 / 布尔 /null/undefined）：
   - 用`typeof`：可区分string/number/boolean/undefined/object（null / 数组 / 对象均返回 object）；
   - 特殊处理 null：`data === null`（单独判断，因为`typeof null === 'object'`是 JS 历史 bug）。
2. 复杂类型判断（数组 / 对象）：
   - 数组：`Array.isArray(data)`（最准确，可区分数组和普通对象）；
   - 普通对象：排除数组 /null 后，`typeof data === 'object'`即可判定。
3. 特殊场景（日期 / 正则等）：
   若后端返回特殊格式（如日期字符串、JSON 格式的正则），需结合业务判断：
   - 日期：通过格式（如2026-03-12）或`new Date(data)`是否有效判断；
   - 枚举 / 特殊标识：按后端接口文档约定（如status: 1/2代表不同状态）。

#### 二、实战流程（前端对接接口时）
1. 先看接口文档：优先按文档定义的类型（如userName: string、list: array）预判，是最高效的方式；
2. 接口返回后验证：
   - 解析 JSON 后，先用`console.log/typeof/Array.isArray`快速校验类型是否符合预期；
   - 封装通用判断函数（如isObject/isArray），避免重复代码；
3. 容错处理：若后端返回类型不符（如本该是数组却返回 null），前端需做默认值兜底（如`const list = res.data || []`）。

### 3. 前后端是怎么连接的
通过**RESTful API** 交互，前端编写get、post、put等请求函数发起HTTP请求，后端处理请求并返回JSON数据。

### 4. 展开运算符
展开运算符（`...`）实现的是**浅拷贝**，仅复制最外层数据，若拷贝的是引用类型，修改拷贝后的数据会影响原数据（如`const b={...a}`，修改`b.name`会同步改变`a.name`）；
**解决深拷贝问题**：使用`structuredClone`方法实现深拷贝，复制多层嵌套的引用类型数据，修改拷贝后的数据不会影响原数据。

### 5. vue3的生命周期
把组件当成一个人，生命周期就是它 出生 → 上班 → 退休 → 去世 的全过程，Vue3 会在每个节点提供钩子函数编写代码，分两种写法，日常开发优先用组合式 API（setup）：

#### 一、组合式 API（Vue3 主流写法｜必背）
直接在 `<script setup>` 里使用，4 个核心钩子即可满足大部分需求：
1. onMounted（最重要）：组件刚显示到页面上；用途：发请求拿数据、初始化定时器
2. onUpdated：数据改了，页面刷新完；用途：数据更新后做后续操作
3. onBeforeUnmounted：组件要离开页面、即将销毁；用途：清定时器、解绑事件（防 bug）
4. onUnmounted：组件彻底消失。

#### 二、选项式 API（兼容 Vue2｜了解）
名字和 Vue2 几乎一样，仅修改最后一个钩子：
- created：组件刚出生（拿数据）
- mounted：页面渲染完成（核心）
- updated：页面更新完成
- beforeUnmount / unmounted：销毁（清垃圾）

### 6. vuex是什么
vuex是Vue项目的**全局共享数据仓库**，类比“全班同学公用的文具盒”，解决多组件之间的数据共享问题，核心包含5个部分：
- state(数据)：存储全局共享数据的地方；
- getters（看数据）：对state中的数据进行加工处理后返回；
- mutations(修改数据的唯一通道)：唯一能修改state数据的地方，必须同步执行；
- actions(干活的)：处理异步逻辑（如调接口），异步完成后调用mutations修改state；
- modules：将大仓库拆分为小仓库，避免数据过多混乱。

### 7. 父子组件通信
- 父传子：通过 **props** 向子组件传值；
- 子传父：通过 **emit** 向父组件发送自定义事件；
- 父调用子方法：Vue3 可通过 **ref + defineExpose** 实现；
- 核心原则：日常开发主要使用 props 和 emit，Vue3 下对应使用 `defineProps` 和 `defineEmits`，数据归父组件管理，遵循单向数据流。

### 8. 什么是 Composition API？为什么要用？
Composition API 是 Vue3 的**组合式写法**，是现在 Vue 开发的标准方式；
优势：代码更整洁、逻辑更好复用、组件维护更方便，解决了Vue2选项式API中复杂组件逻辑分散的问题。

### 9. setup 是什么时候执行的？
- 执行时机：组件`beforeCreate`钩子前立即执行，仅执行一次； 
- 核心限制：内部无`this`，需通过参数获取props、emit等； 
- 核心价值：作为组合式 API 的入口，集中管理组件的响应式数据、方法和业务逻辑。

### 10. ref 和 reactive 有什么区别？
- 适用类型不同：ref 更适合处理基本数据类型（数字、字符串、布尔等），也可包裹对象 / 数组；reactive 仅能处理复杂数据类型（对象、数组），无法直接绑定基本类型。 
- 访问方式不同：ref 创建的响应式数据，需通过`.value`访问 / 修改（模板中可省略）；reactive 创建的对象，可直接访问属性（无需`.value`）。 
- 响应式原理 / 特性不同：ref 通过包装成RefImpl对象实现响应式，能保留基本类型的响应性；reactive 基于 Proxy 代理对象实现，深层嵌套属性也具备响应性，但重新赋值会丢失响应性（需用 Object.assign 替换）。

### 11. Vue3 中父子组件如何通信？
日常开发中父子通信主要使用 **props 和 emit**，Vue3 下在`<script setup>`中使用 `defineProps` 接收父组件传值，使用 `defineEmits` 触发自定义事件向父组件传值。

### 12. computed vs watch
| 特性         | computed（计算属性）| watch（监听器）|
|--------------|---------------------------------------------|---------------------------------------------|
| 核心用途     | 依赖其他响应式数据衍生新值，侧重数据推导/转换 | 监听数据变化执行自定义逻辑，侧重副作用操作   |
| 使用形式     | 函数形式返回结果，**有缓存**                 | 对象/函数形式，**无缓存**                   |
| 返回值       | 必须有返回值，可直接在模板/代码中使用       | 无强制返回值，仅执行回调逻辑                |
| 配置项       | 无                                          | 可配置immediate（初始化执行）、deep（深度监听）|
| 触发时机     | 依赖数据变化时触发                          | 监听数据变化时立即触发                      |

**简单总结**：需要推导新数据用computed，需要监听数据做异步/DOM操作用watch。

### 13. vue路由
Vue 路由（`vue-router`）是单页面应用（SPA）的 **“页面导航员”**，核心作用是**不刷新浏览器，仅切换页面内的组件**（SPA 整个项目只有一个 HTML 文件）。

#### 核心使用步骤（3步）
1. 定义 “路径 - 组件” 的匹配规则（如 `/home` 对应首页组件）；
2. 在页面中放置 `<router-view>` 标签：作为“组件展示区”，路由匹配到的组件会渲染在此处；
3. 实现页面跳转：用 `<router-link>` 标签（替代a标签，声明式导航）或代码中`router.push()`（编程式导航），跳转后`<router-view>` 内组件自动切换。

#### 常用功能
- 动态路由：如 `/detail/:id`，可匹配 `/detail/1`、`/detail/2`，实现不同参数的页面复用（如商品详情页）；
- 路由守卫：如未登录时拦截跳转到个人中心的请求，实现权限控制，相当于“门禁”。

#### 核心总结
1. Vue 路由核心是 “路径和组件的映射”，实现单页面无刷新切换；
2. 核心标签：`<router-view>`（展示组件）、`<router-link>`（声明式跳转）；
3. 常用功能：动态路由（适配不同参数页面）、路由守卫（权限控制）、编程式导航（`router.push`）。

### 14. vue组件的通信方式有几种？
Vue组件通信分多种场景，核心通信方式如下：
#### 父子组件通信（最常用）
- 父传子：Props 传递数据、Provide/Inject 跨层级传递；
- 子传父：emit 触发自定义事件、v-model 实现双向绑定；
- 父调用子：refs 获取子组件实例（慎用，破坏组件封装性）。

#### 跨组件/全局通信
- Vuex/Pinia：全局状态管理，解决多组件跨层级数据共享问题；
- 事件总线：适用于小型项目，通过自定义事件实现任意组件通信；
- localStorage/sessionStorage：通过本地存储实现跨页面通信（非响应式）。

#### 核心原则
遵循**‘单向数据流’**原则：父组件通过 Props 传递数据，子组件通过事件通知父组件修改数据；复杂场景用 Vuex/Pinia 统一管理状态。

### 15. vuex的核心使用
Vuex 是 Vue 项目里的 **“全局数据仓库”**，解决多个组件（如首页、购物车、个人中心）共享数据的问题，避免组件间来回传值的繁琐。

#### 核心5个部分（关键作用）
1. state：存数据的地方（如`state.userInfo`存用户信息），相当于仓库里的 “货架”；
2. mutations：唯一能改 state 数据的地方，必须**同步执行**，相当于 “改数据的唯一入口”；
3. actions：处理**异步逻辑**（如调接口拿用户信息），异步完成后再调用 mutations 改 state，相当于 “异步操作的中转站”；
4. getters：对 state 里的数据做加工（如从购物车列表筛选已选中商品），相当于 “数据加工器”；
5. modules：把大仓库拆成小仓库（如用户模块、购物车模块），避免数据太多混乱。

#### 核心使用规则
组件中**不能直接修改state**，修改流程：组件调actions（异步逻辑）→ actions调mutations → mutations修改state；
组件获取数据：直接从state取值，或通过getters加工后取值。

#### 适用场景
- 中大型项目：推荐使用Vuex管理全局共享数据；
- 小型项目：使用简单的全局变量即可，无需Vuex。

### 16. computed和watch的使用场景
- computed：**计算并返回一个新的值**，适用于需要根据现有响应式数据推导新数据的场景（如拼接用户名、计算购物车总价、数据格式化）；
- watch：**监听数据变化，执行副作用操作**，适用于数据变化后需要做异步请求、修改DOM、操作定时器等场景（如监听表单字段变化做实时校验、监听路由变化做页面逻辑处理）。

### 17. VUE项目里会用到路由吗？
**90%以上的Vue项目必用路由**，极少数纯单组件页面场景不用，具体分场景说明：

#### 一、必用路由的场景（SPA 核心场景）
如果是**单页面应用（SPA）**（Vue 最核心的应用场景），路由是必备的：
- SPA 核心是 “页面不刷新，仅切换组件”，路由通过 URL 路径匹配对应组件，渲染到`<router-view>`容器中，实现首页、列表页、详情页等不同页面的切换；
- 典型场景：后台管理系统、电商 H5、移动端 App 内嵌页面等，依赖路由做页面导航、权限控制（如登录拦截）、参数传递（如详情页 id）。

#### 二、几乎不用路由的场景（极少数）
仅当项目是**纯单组件页面**（无页面切换需求）时，比如：
- 简单的表单页、静态展示页、独立的组件封装库；
- 这类场景只需单个 Vue 实例挂载，无需多页面切换，因此不用路由。

#### 三、Vue 路由的核心使用价值
1. 实现 SPA 的无刷新页面切换，提升用户体验；
2. 支持动态路由（如/user/:id）、嵌套路由（如侧边栏 + 内容区布局）；
3. 提供路由守卫（如beforeEach）做权限控制、页面跳转拦截；
4. 支持编程式导航（router.push）和声明式导航（<router-link>），满足不同跳转需求。

### 18. 发请求时实现按钮禁用+加载动画，请求结束后恢复
**核心逻辑**：发起请求前修改状态禁用按钮并显示loading；请求结束后（无论成功/失败）重置状态，恢复按钮并关闭loading。
```vue
<template>
  <button @click="fetchData" :disabled="loading">
    <span v-if="loading">加载中...</span>
    <span v-else>发起请求</span>
  </button>
</template>

<script setup>
import { ref } from 'vue'
import axios from 'axios'

const loading = ref(false)
const fetchData = async () => {
  // 请求前：禁用按钮+显示loading
  loading.value = true
  try {
    const res = await axios.get('/api/data')
    console.log(res.data)
  } catch (err) {
    console.error(err)
  } finally {
    // 请求结束（成功/失败）：恢复按钮+关闭loading
    loading.value = false
  }
}
</script>
```

### 19. v-if 和 v-show 的区别
v-if 和 v-show 都是用来控制元素显示隐藏的，核心区别有3点：
1. **控制方式不同**
   - v-if：直接**创建 / 删除 DOM 节点**（真正的添加 / 移除）
   - v-show：通过 CSS 的`display:none`控制显示隐藏（DOM节点一直存在于页面中）
2. **切换性能不同**
   - v-if：切换开销大（每次切换都要重建/删除DOM）
   - v-show：切换开销小（仅修改CSS样式，无DOM操作）
3. **使用场景不同**
   - v-if：**不频繁切换**、条件复杂时使用（如根据用户权限显示/隐藏模块）
   - v-show：**频繁切换**（如点击按钮显示/隐藏弹窗、下拉菜单）时使用

### 20. vue中的watch怎么用？
watch 是 Vue 提供的**监听器**，用来监听响应式数据的变化，数据一旦改变，会自动触发对应的回调函数。
#### 核心特性
- 能获取数据的**新值和旧值**；
- 可配置`immediate`：页面初始化时立即执行一次回调；
- 可配置`deep`：深度监听对象/数组的嵌套属性变化。

#### 基础使用示例（Vue3 组合式API）
```vue
<script setup>
import { ref, watch } from 'vue'

// 监听基本类型
const num = ref(0)
watch(num, (newVal, oldVal) => {
  console.log(`num从${oldVal}变成了${newVal}`)
})

// 监听对象（深度监听）
const user = ref({ name: '张三', age: 20 })
watch(user, (newVal) => {
  console.log('user信息变化：', newVal)
}, { deep: true, immediate: true })
</script>
```

### 6. es6如何向后端传输数据
ES6 向后端传数据的本质是**通过 HTTP 请求将数据发送到后端接口**，核心遵循“按需选择请求方式和数据格式”原则，具体规则：
- 查数据：用 **GET** 请求，参数拼接在URL上（简单数据）；
- 创/改/删数据：用 **POST/PUT/DELETE** 请求，参数放在请求体中（更安全）；
- 普通数据：用 **JSON** 格式（前后端通用，ES6 可通过`JSON.stringify`转换），请求头设置`Content-Type: application/json`；
- 文件/二进制数据：用 **FormData** 格式，支持文件上传，无需手动转换格式；
- 常用请求库：axios/fetch（基于Promise，适配ES6异步语法）。

### 7. 前后端的连接流程
前后端通过**HTTP/HTTPS协议**完成数据交互，核心流程为：
1. 浏览器（前端）：发起 HTTP 请求（GET/POST等），携带请求参数、请求头；
2. 服务器（后端）：接收请求，处理业务逻辑（如操作数据库、计算数据）；
3. 服务器（后端）：处理完成后，返回 JSON 格式的响应数据（含状态码、响应体）；
4. 浏览器（前端）：解析返回的 JSON 数据，渲染到页面上（或做后续逻辑处理）。

### 8. 什么是 RESTful API ？
RESTful API 是一种**基于 REST 架构风格设计的 Web 接口**，是前后端交互的主流接口规范，核心设计原则：
1. 用**HTTP方法表示操作**，避免在URL中出现动词（如用POST /user表示创建用户，而非GET /createUser）；
2. 用**URL表示资源**，资源名用名词复数形式（如/users、/products）；
3. 接口**无状态**：每个请求包含服务器处理所需的所有信息，服务器不存储客户端状态；
4. 返回**标准HTTP状态码**（200成功、201创建、404未找到、500服务器错误等）；
5. 支持**版本控制**：在URL中标识接口版本（如/api/v1/users）；
6. 支持**分页/筛选**：通过请求参数实现数据分页（如?page=1&size=10）、筛选（如?status=1）。

### 9. 常见的HTTP状态码
#### 成功响应（2xx）
- 200：请求成功，服务器正常返回数据；
- 201：已创建，请求成功并创建了新的资源（如POST创建用户）；
- 204：无内容，服务器处理成功，但不返回任何响应体。

#### 重定向响应（3xx）
- 304：未修改，资源未发生变化，浏览器使用本地缓存数据。

#### 客户端错误（4xx）
- 400：请求格式错误，服务器无法解析请求参数；
- 401：未授权，需要身份认证（如未登录）；
- 403：禁止访问，已认证但无权限操作资源；
- 404：资源未找到，请求的URL不存在。

#### 服务器错误（5xx）
- 500：服务器内部错误，服务器处理请求时发生未知错误；
- 502：网关无效响应，服务器作为网关/代理，收到了无效的响应。

### 10. 常见的HTTP方法
HTTP方法代表对资源的操作，RESTful API 严格遵循方法的语义，核心方法：
- GET：**查询**资源，请求参数拼在URL，可缓存，无请求体；
- POST：**创建**资源，请求参数放在请求体，不可缓存；
- PUT：**更新**资源，全量更新，需传入资源的所有属性；
- DELETE：**删除**资源，删除指定URL的资源；
- PATCH：**局部更新**资源，仅传入需要修改的属性（补充PUT）。

### 11. TCP协议与UDP协议的区别
| 特性         | TCP 协议                | UDP 协议                |
|--------------|-------------------------|-------------------------|
| 连接性       | 面向连接（三次握手建立连接） | 无连接（无需建立连接，直接发数据） |
| 可靠性       | 可靠传输（重传丢失的数据包、排序、校验） | 不可靠传输（不保证数据到达，无重传） |
| 性能         | 速度慢，开销大（维护连接、可靠性） | 速度快，开销小（无额外控制逻辑） |
| 适用场景     | 对可靠性要求高的场景（HTTP/HTTPS、文件传输、聊天） | 对实时性要求高的场景（视频直播、语音通话、游戏） |

## JS
### 1. var let const区别
- var：全局/函数作用域，存在**变量提升**、**重复声明**，后声明可覆盖先前的，现代开发极少使用；
- let：块级作用域，无变量提升、不允许重复声明，用于声明**需要后续修改值**的变量；
- const：块级作用域，无变量提升、不允许重复声明，声明时**必须初始化**，不能修改变量的指向（基本类型值不可改，对象/数组可修改内部属性），用于声明**无需修改的常量**；
- 开发共识：**const 优先，let 次之，var 不用**。

### 2. map,set,array的区别
三者均为JS的集合类数据结构，核心区别和适用场景：
- Array（数组）：**有序、可重复**的集合，通过索引访问元素，是前端最常用的列表存储结构；
- Set（集合）：**无序、不重复**的集合，无索引，适合**去重、判断元素是否存在**等场景；
- Map（映射）：**键值对结构**，键可以是任意类型（不限于字符串），**有序、键不重复**，适合**存储键值映射关系、查关系**等场景。

**口语化总结**：去重用 Set，查关系用 Map，普通列表用 Array。

### 3. js的各种作用域和作用域链
#### 作用域
作用域是**变量的可访问范围**，决定了变量在哪个区域能被读取/修改，JS主要有3种作用域：
1. 全局作用域：变量在整个脚本文件中可访问，挂载在window对象上（如全局声明的var、未声明的变量）；
2. 函数作用域：变量仅在函数内部可访问（如函数内声明的var）；
3. 块级作用域：变量仅在代码块（{}）内可访问（let/const 声明的变量）。

#### 作用域链
作用域链是**变量的查找规则**，当访问一个变量时，JS引擎会按**由内向外**的顺序在当前作用域、父级作用域、全局作用域中查找，直到找到变量或报错；
**核心规则**：
- 词法作用域（静态作用域）：作用域在**定义时**确定，而非执行时；
- 查找方向：由内向外，不会反向查找（子作用域可访问父作用域，父作用域不可访问子作用域）。

### 4. 暂时性死区（TDZ）
暂时性死区（TDZ）是**let/const**的特性，指：**在let/const声明变量之前，变量虽然已经存在于作用域中，但无法被访问/使用，访问即报错**，这段“声明前不可访问”的区域就是暂时性死区。

#### 与var的对比
- var：存在变量提升，声明前访问变量会得到`undefined`，无暂时性死区；
- let/const：存在**变量提升但不初始化**，声明前处于暂时性死区，访问直接报错。

**简单总结**：TDZ = let/const 声明前用就死，var 声明前用是undefined。

### 5. 函数定义的方式
JS有3种核心的函数定义方式，各有特点和适用场景：
1. **函数声明**：`function 函数名() { ... }`
   - 特点：有函数名，存在**变量提升**，可在声明前调用；
   - 适用：全局/函数内的普通函数定义。
2. **函数表达式**：`const 变量名 = function() { ... }`（可匿名/具名）
   - 特点：赋值给变量，无变量提升，**必须先定义再调用**；
   - 适用：作为回调函数、赋值给变量的函数。
3. **箭头函数（ES6）**：`const 变量名 = () => { ... }`
   - 特点：写法简洁，**无自己的this**（继承外层作用域的this），不能作为构造函数；
   - 适用：回调场景（如数组遍历、Promise回调）、简化简单函数。

### 6. 事件绑定的方式
JS事件绑定主要有3种方式，按推荐度和特性划分：
1. **行内绑定**：直接在HTML标签中写on+事件名（如`<button onclick="fn()">`）；
   - 特点：耦合度高（JS和HTML混写），不推荐使用；
2. **DOM0级绑定**：通过元素对象赋值（如`btn.onclick = fn`）；
   - 特点：简单，**同一事件只能绑定一个处理函数**，后续绑定会覆盖先前的；
3. **DOM2级绑定**：通过`addEventListener`绑定（如`btn.addEventListener('click', fn)`）；
   - 特点：**同一事件可绑定多个处理函数**，支持事件捕获/冒泡，可通过`removeEventListener`解绑，**推荐使用**。

### 7. 一个等于号、两个等于号、三个等于号的区别
三者均为JS的赋值/比较运算符，核心语义完全不同：
1. **一个等于号（=）**：**赋值运算符**，将右侧值赋值给左侧变量（如`let a = 1`）；
2. **两个等于号（==）**：**松散相等**，判断值是否相等，**会进行自动类型转换**（如`1 == '1'`为true），存在诸多坑，不推荐使用；
3. **三个等于号（===）**：**严格相等**，判断值和类型是否都相等，**不进行类型转换**（如`1 === '1'`为false），日常开发**优先使用**。

### 8. 两个等号（==）的类型转换规则
`==` 执行**松散相等判断**，核心规则是**先转类型，再比较值**，核心转换逻辑：
1. 字符串 / 布尔值 与 数字比较：字符串、布尔值**转为数字**后比较（如`'2' == 2`→`2==2`，`true == 1`→`1==1`）；
2. null 和 undefined 比较：`null == undefined`为true，与其他类型比较均为false；
3. 对象与基本类型比较：对象**转为基本类型**（调用valueOf/toString）后比较（如`[1] == 1`→`'1'==1`→`1==1`）；
4. NaN 与任何值比较（包括自身）：均为false。

**开发建议**：日常开发**优先使用===**，避免`==`的类型转换坑。

### 9. 事件冒泡是什么，怎么解决
#### 事件冒泡是什么？
事件冒泡是浏览器的**默认事件传播机制**，指：当元素触发一个事件时，事件会从**触发的子元素**开始，**逐层向上传播**到父元素、祖父元素，直到根节点（document），即**子 → 父 → 爷爷 一路传**。

#### 怎么解决事件冒泡？
通过**阻止事件传播**解决，核心方法：
- 原生JS：在事件回调中调用`event.stopPropagation()`；
- Vue中：使用事件修饰符`.stop`（如`@click.stop="fn()"`）。

#### 什么时候不解决？
**事件委托**场景下，故意利用事件冒泡：将事件绑定在父元素上，通过冒泡机制处理子元素的事件，减少事件绑定次数，提升性能（如列表项的点击事件，绑定在列表父容器上）。

#### 实际开发问题示例
卡片内有删除按钮，点击删除按钮时，事件冒泡会导致卡片的点击跳转事件也被触发，造成误触；此时需在删除按钮的点击事件中阻止冒泡。

#### 核心总结
事件冒泡不是bug，是浏览器的**原生特性**：
- 不想要冒泡 → 用`stopPropagation`/.stop掐断；
- 想要利用冒泡 → 做事件委托，提升性能。

### 10. 除了a标签，js实现页面跳转的方式
页面跳转除了使用 `<a>` 标签，还可通过**操作window.location对象**实现，核心3种方式：
1. **location.href**：最常用，跳转到指定URL，**保留浏览器历史记录**（可点击返回键回到原页面）；
   ```js
   window.location.href = 'https://www.baidu.com'
   ```
2. **location.assign()**：与href功能一致，跳转到指定URL，**保留历史记录**；
   ```js
   window.location.assign('https://www.baidu.com')
   ```
3. **location.replace()**：跳转到指定URL，**不保留浏览器历史记录**（无法点击返回键回到原页面）；
   ```js
   window.location.replace('https://www.baidu.com')
   ```
4. 补充：**location.reload()**：刷新当前页面，非跳转，但属于常用的location操作。

### 11. Promise的作用和使用
#### Promise的作用
Promise 是 JS 用于处理**异步操作**的对象，核心解决了**回调地狱**问题（多层异步回调嵌套导致代码混乱、难以维护），主要用途：
1. 统一异步操作的处理规范，让异步代码更易读、易维护；
2. 支持多个异步操作的串行/并行执行（如`Promise.all`、`Promise.race`）；
3. 实现错误的统一捕获（通过`catch`捕获整个异步流程的错误）；
4. 为async/await（ES7）提供底层支持，让异步代码“看起来像同步代码”。

#### 常用场景
前端接口请求（axios/fetch 均基于Promise）、定时器异步操作、文件读取等异步场景。

#### 基础使用
```js
// 创建Promise
const p = new Promise((resolve, reject) => {
  // 异步操作（如调接口）
  setTimeout(() => {
    const res = { code: 200, data: '成功' }
    if (res.code === 200) {
      resolve(res.data) // 异步成功，调用resolve
    } else {
      reject('失败') // 异步失败，调用reject
    }
  }, 1000)
})

// 执行Promise
p.then(res => {
  console.log(res) // 接收成功结果
}).catch(err => {
  console.error(err) // 捕获错误
})
```

### 12. 前后端数据通信（fetch的使用）
fetch 是**浏览器原生的API**，用于发起HTTP请求获取后端数据，底层基于Promise，核心特点：**浏览器在后台完成请求发送和响应接收，将结果包装成Promise返回给前端**。

#### 核心注意点
fetch 获取的响应对象**不是数据本身**，而是一个“响应包裹体”，包含状态码、headers、body等信息，需通过`res.json()`解析为JSON数据（相当于“打开外卖盒子拿饭”）。

#### 基础使用示例
```js
// 发起GET请求获取后端数据
fetch('/api/data')
  .then(res => {
    // 先判断请求是否成功（fetch仅在网络错误时拒绝，状态码404/500仍会resolve）
    if (!res.ok) {
      throw new Error(`请求失败：${res.status}`)
    }
    return res.json() // 解析JSON数据，返回新的Promise
  })
  .then(data => {
    console.log('后端返回数据：', data) // 处理真正的业务数据
  })
  .catch(err => {
    console.error('请求错误：', err) // 捕获所有错误（网络错误、解析错误、状态码错误）
  })
```

### 13. 事件循环（Event Loop）
事件循环是JS**单线程**处理异步操作的核心机制，保证异步操作按顺序执行，核心规则：
1. JS执行分为**同步任务**和**异步任务**，同步任务优先执行，在主线程中按顺序执行；
2. 异步任务分为**宏任务**和**微任务**，执行完所有同步任务后，再处理异步任务；
   - 宏任务：setTimeout/setInterval、DOM事件、AJAX请求、script整体代码；
   - 微任务：Promise的then/catch/finally、async/await、MutationObserver；
3. 异步任务执行顺序：**先执行所有微任务，再执行一个宏任务**，执行完宏任务后，再检查并执行所有微任务，如此循环（即“宏任务→微任务→宏任务→微任务...”）；
4. 核心细节：Promise 的回调属于微任务，即使在setTimeout（宏任务）中创建，也会在当前宏任务结束后**立即执行**，优先于下一个宏任务。

### 14. this指向问题
this 是JS的关键字，代表**当前执行上下文的执行主体**，简单理解为**“到底谁在调用函数、谁在干活”**，this的指向**在函数执行时确定**，而非定义时，核心判断规则（按优先级从高到低）：
1. **new绑定**：函数通过new调用（构造函数），this指向新创建的实例对象；
2. **显式绑定**：通过call/apply/bind调用函数，this指向绑定的对象；
3. **隐式绑定**：函数通过对象的方法调用（如`obj.fn()`），this指向**点（.）前面的对象**；
4. **默认绑定**：函数独立调用（无上述绑定），非严格模式下this指向window，严格模式下为undefined；
5. **特殊情况**：箭头函数无自己的this，继承**外层作用域**的this。

### 15. 箭头函数
箭头函数是ES6的函数简写形式，核心特性和使用规则：
1. **写法简洁**：省略function关键字，单参数可省略()，单行返回可省略{}和return；
2. **无自己的this**：箭头函数的this**继承外层作用域的this**，且**一旦确定无法修改**（无法通过call/apply/bind改变）；
3. **无arguments对象**：需通过**剩余参数（...args）** 获取传入的参数；
4. **不能作为构造函数**：不能用new调用，否则会报错；
5. **适用场景**：回调函数（数组遍历、Promise回调）、简化简单函数，避免this指向混乱；
6. **不适用场景**：构造函数、对象的方法、需要动态this的场景。

#### this指向快速判断
看到箭头函数的this，只需判断**外层作用域的this**；
看到普通函数的this，只需问2个问题：
- 有没有 **.** 点出来？（如`obj.say()`）→ this = 点前面的对象；
- 有没有被“单独拿出来”？（如`const fn = obj.say; fn()`）→ this = window/undefined。

### 16. 单引号、双引号、反引号的作用
JS中的单引号、双引号、反引号均用于表示字符串，核心区别：
1. **单引号（' '）**：表示普通字符串，内部嵌套双引号时无需转义（如`'hello "world"'`）；
2. **双引号（" "）**：表示普通字符串，内部嵌套单引号时无需转义（如`"hello 'world'"`）；
3. **反引号（` `）**：ES6新增的**模板字符串**，功能最强大，核心特性：
   - 支持**变量插值**：通过`${变量/表达式}`将变量/表达式嵌入字符串（如`Hello ${name}`）；
   - 支持**多行字符串**：直接换行书写，无需拼接换行符\n；
   - 支持**表达式运算**：${}内可直接执行JS表达式（如`1+2=${1+2}`）。

**Java与JS的区别**：Java中单引号表示**字符类型**（单个字符），双引号表示**字符串类型**；JS中无字符类型，单/双引号均表示字符串。

### 17. 模板字符串 + 解构 + 箭头函数 = 现代前端最常见三件套
这三个均为ES6核心语法，是现代前端开发的**高频基础标配**，三者结合让代码更简洁、高效、易维护：
- **模板字符串**：用`${}`简化字符串拼接，替代传统的`+`拼接，支持多行和表达式；
- **解构赋值**：快速从对象/数组中提取数据，无需逐个赋值（如`const { name, age } = user`）；
- **箭头函数**：精简函数写法，无自己的this，避免异步回调中的this指向混乱；

#### 组合使用示例
```js
const user = { name: '张三', age: 20 }
// 解构 + 箭头函数 + 模板字符串
const getUserInfo = ({ name, age }) => `姓名：${name}，年龄：${age+1}`
console.log(getUserInfo(user)) // 姓名：张三，年龄：21
```

### 18. 子元素触发事件会不会影响到父元素，为什么？
**会影响**。
**原因**：浏览器的**事件冒泡机制**——子元素触发事件后，事件会从子元素开始，逐层向上传播到父元素、祖父元素，直到根节点，父元素的同名事件会被触发。
**解决方法**：若不想父元素被影响，在子元素的事件回调中**阻止事件冒泡**（原生JS用`event.stopPropagation()`，Vue用`.stop`修饰符）。
**简单总结**：子点父会响，原因是冒泡；不想父知道，用stopPropagation。

### 19. js数据加密、解密讲解
前端JS**很少自行实现数据加密解密**，核心原则和实际方案：
1. **传输过程加密**：核心依赖**HTTPS协议**，HTTPS通过SSL/TLS对传输的数据进行加密，防止数据被劫持、篡改，这是前后端数据传输安全的**核心保障**；
2. **真正的加解密在后端**：前端仅需将明文数据发送给后端，由后端进行加密（如MD5、SHA256、RSA）存储；后端返回的数据也由后端解密，前端无需处理；
3. **前端仅做简单处理**：前端可做简单的哈希处理（如MD5），但哈希是**不可逆的**（能还原是加密，不能还原是哈希），主要用于密码防明文传输（但仍需配合HTTPS）；
4. **核心禁忌**：前端**不存储加解密的密钥**，否则容易被破解，失去加密意义。

**总结**：前端JS无需深入做加解密，HTTPS是传输安全的关键，真正的加解密逻辑放在后端实现。

### 20. js通过浏览器怎么获取json并解析
浏览器中JS获取并解析JSON数据主要有2种方式，**现代浏览器首选fetch**，老浏览器用XHR，核心均为“发起请求→获取JSON字符串→解析为JS对象”。

#### 一、现代浏览器首选（fetch 方式）
fetch 是浏览器原生API，基于Promise，自动处理部分解析逻辑，步骤简单：
1. 调用`fetch()`发起GET请求，指向JSON文件/接口地址；
2. 调用响应对象的`json()`方法，**自动将JSON字符串解析为JS对象**；
3. 处理解析后的JS对象，通过`catch`捕获请求/解析错误。
```js
// 获取并解析JSON
fetch('data.json') // 本地JSON文件/后端接口地址
  .then(res => {
    if (!res.ok) throw new Error('请求失败')
    return res.json() // 自动解析JSON为JS对象
  })
  .then(data => {
    console.log('解析后的数据：', data) // 直接使用JS对象
  })
  .catch(err => console.error('错误：', err))
```

#### 二、老浏览器兼容（XHR 方式）
XMLHttpRequest（XHR）是传统的请求方式，需手动解析JSON，步骤稍繁琐：
1. 创建XHR对象：`new XMLHttpRequest()`；
2. 打开请求：`xhr.open('GET', 'data.json', true)`（异步请求）；
3. 发送请求：`xhr.send()`；
4. 监听请求完成事件，通过`xhr.responseText`获取JSON字符串；
5. 用`JSON.parse()`**手动解析JSON字符串为JS对象**。
```js
// 创建XHR对象
const xhr = new XMLHttpRequest()
// 打开请求
xhr.open('GET', 'data.json', true)
// 监听请求状态变化
xhr.onreadystatechange = function() {
  // 请求完成且成功
  if (xhr.readyState === 4 && xhr.status === 200) {
    const jsonStr = xhr.responseText // 获取JSON字符串
    const data = JSON.parse(jsonStr) // 手动解析为JS对象
    console.log('解析后的数据：', data)
  }
}
// 发送请求
xhr.send()
```

### 21. js的JSON形式出问题了，如何把数据发送到后端
JSON格式出问题时，核心思路是**先排查问题，再替换传输方式**，步骤和解决方案：
1. **第一步：排查JSON问题**
   确认数据是否为**合法JSON**，能否通过`JSON.stringify()`转换为JSON字符串（若转换报错，说明数据存在循环引用、特殊值等问题）；
   - 解决：清理数据中的循环引用、删除undefined/function等JSON不支持的类型。

2. **第二步：检查请求头**
   若手动设置请求头，确认是否添加了`Content-Type: application/json`，缺失会导致后端无法解析JSON数据；
   - 解决：补充正确的请求头。

3. **第三步：替换数据传输方式**
   若上述方法无法解决，直接**替换为FormData格式**发送数据（FormData无需JSON转换，支持所有数据类型，适合文件/复杂数据传输）；
   ```js
   // 用FormData发送数据，无需JSON转换
   const formData = new FormData()
   formData.append('name', '张三')
   formData.append('age', 20)
   // 发起请求
   axios.post('/api/user', formData)
   ```

**核心总结**：先看`JSON.stringify`能否转换，再检查Content-Type请求头，不行就换FormData格式。

### 22. js有哪些数据类型？
JS的数据类型分为**基本数据类型（值类型）** 和**引用数据类型（复杂类型）**，ES6及后续新增了Symbol和BigInt，目前共8种核心数据类型。

#### 一、基本数据类型（7 种，存储在栈中）
基本数据类型为**值类型**，直接存储值，赋值时进行**值拷贝**，修改拷贝后的值不影响原数据：
1. Number：数字（整数、小数、NaN、Infinity）；
2. String：字符串（单 / 双 / 反引号包裹的文本）；
3. Boolean：布尔值（true/false）；
4. Undefined：声明变量未赋值时的默认值；
5. Null：表示空值（手动赋值的 “空”，`typeof null === 'object'`是JS历史bug）；
6. Symbol（ES6 新增）：唯一且不可变的值，用于对象的唯一属性名；
7. BigInt（ES11 新增）：表示超大整数（解决Number的精度限制，数字后加后缀n，如`100n`）。

#### 二、引用数据类型（1 种，存储在堆中）
引用数据类型为**复杂类型**，栈中存储地址，堆中存储实际数据，赋值时进行**地址拷贝**，修改拷贝后的值会影响原数据：
- Object：包含普通对象、数组（Array）、函数（Function）、日期（Date）、正则（RegExp）等，本质都是Object的衍生类型。

### 23. async/await 是promise的语法糖
async/await 是**ES7新增的异步语法**，本质是**Promise的语法糖**，**不增加新的功能**，仅让Promise的异步代码**书写更规范、更像同步代码**，核心特性：
1. **async**：修饰函数，async函数的**返回值一定是Promise对象**（若返回非Promise值，会自动包装为Promise.resolve(值)）；
2. **await**：只能在**async函数内部使用**，用于“等待”Promise的执行结果，**会暂停当前async函数的执行**，直到Promise状态变为resolved/rejected，再继续执行后续代码；
3. **错误处理**：通过**try/catch**捕获await的错误（替代Promise的catch）；
4. **底层原理**：基于Promise和生成器（Generator）实现，浏览器/Node.js底层仍按Promise的事件循环执行。

#### 对比示例（Promise vs async/await）
```js
// Promise方式
const fn1 = () => {
  axios.get('/api/1')
    .then(res1 => axios.get(`/api/2?${res1.data.id}`))
    .then(res2 => console.log(res2.data))
    .catch(err => console.error(err))
}

// async/await方式（更简洁，像同步代码）
const fn2 = async () => {
  try {
    const res1 = await axios.get('/api/1')
    const res2 = await axios.get(`/api/2?${res1.data.id}`)
    console.log(res2.data)
  } catch (err) {
    console.error(err)
  }
}
```

### 24. mvvm是什么
MVVM 是**前端的架构模式**，是Model-View-ViewModel的缩写，将页面分为三个核心部分，实现**数据和视图的双向绑定**，是Vue/React等框架的核心设计思想：
1. **M（Model，模型）**：表示**数据层**，即页面的响应式数据（如Vue中的data、ref），仅存储数据，不关心视图和业务逻辑；
2. **V（View，视图）**：表示**视图层**，即页面的HTML/CSS结构，负责展示数据，是用户能看到的界面；
3. **VM（ViewModel，视图模型）**：表示**中间层**，连接Model和View的桥梁，核心作用：
   - 数据绑定：将Model中的数据同步到View中，数据变化时视图自动更新；
   - 事件监听：监听View中的用户操作（如点击、输入），触发事件后更新Model中的数据；
   - 实现**数据和视图的双向绑定**，开发者只需关注数据变化，无需手动操作DOM。

**核心优势**：分离数据和视图，减少DOM操作，提高开发效率，让代码更易维护。

### 25. JS声明变量的关键字
JS声明变量主要有**三个关键字**：var、let、const，其中let和const是ES6新增，替代var成为现代开发的主流，三者核心区别见**JS-1. var let const区别**，核心说明：
1. **var**：ES5及更早的声明方式，函数级作用域，存在变量提升、允许重复声明，现代开发中因易引发bug已极少使用；
2. **let**：ES6 新增，块级作用域，无变量提升，不允许重复声明，用于声明**需要后续修改值**的变量；
3. **const**：ES6 新增，块级作用域，无变量提升，声明时**必须初始化**，且不能修改变量的指向（基本类型值不可改，对象 / 数组可修改内部属性），用于声明**无需修改的常量**。

**开发共识**：**const 优先，let 次之，var 不用**。
